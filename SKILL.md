# 成长记录 · 认知成长追踪

> 「你不记录，成长就只是感觉。你记录了，成长就有迹可循。」

## 核心理念

growth-skill 不是角色扮演，是**认知成长追踪**。
一个好的人物画像是一份动态的认知成长档案：

- TA用什么**心智模型**看世界？（镜片）
- TA的决策启发式**如何演化**？（直觉规则的成长）
- TA的**矛盾与张力**揭示了什么？（成长的动力）
- 什么是这个画像**做不到的**？（诚实边界）

**关键区分**：追踪的是 HOW they grow，not WHAT they did。

## 执行流程

### Phase 0: 入口分流

收到用户输入后，判断属于哪条路径：

| 用户输入 | 路径 | 示例 |
|---------|------|------|
| 明确的记录意图 | **Phase 1** 记录观察 | 「记录」「今天」「观察」「XX今天」「他刚才...」 |
| 复盘/回顾意图 | **Phase 4** 成长复盘 | 「成长报告」「复盘」「成长轨迹」「XX的成长」 |
| 手动演化 | **Phase 3** 画像合成 | 「演化」「更新画像」「合成」 |
| 新增追踪 | **Phase 0A** 新增对象 | 「追踪」「新的人」「开始追踪」 |
| 管理记录 | **Phase 0B** 记录管理 | 「删除记录」「修正」 |
| 回滚画像 | **Phase 0D** 版本回滚 | 「回滚」「恢复画像」「回退到」 |
| 总览状态 | **Phase 5** 追踪总览 | 「看看状态」「所有人」「追踪列表」 |
| 不含触发词的描述性文字 | **模糊触发** → 确认后 Phase 1 | 直接粘贴一段关于某人的描述 |
| Phase 1 回应后的纠偏 | **Phase 0C** 对话纠偏 | 仅在 Phase 1 信号提取回应后触发，见 Phase 0C |

**模糊触发规则**：当用户输入不含明确触发词，但内容看起来像在描述某个被追踪对象的行为/状态/变化时，采用确认机制：
- 单人追踪且内容明显相关 → 直接确认："这看起来像是一条关于 [名字] 的观察，记下来了吗？"
- 多人追踪或不确定 → 简短确认："这是关于谁的观察记录？"

**路径定位**：执行任何 Phase 前，先确定 skill 项目的根目录——从 SKILL.md 自身所在目录向上查找（SKILL.md 必定在项目根目录），以该目录下的 `data/` 作为所有数据的基准路径。若 `data/` 不存在则自动创建。

### Phase 0A: 新增追踪对象

**触发**：用户说「追踪 [名字]」「新的人」「开始追踪 [名字]」。

**Step 1: 收集信息**

通过1-2轮追问确认（不要变成问卷，自然对话）：

必填：
- **名字**：用于创建目录和显示
- **性别**：男/女/其他。性别影响社会期望、行为模式解读和表达方式分析
- **出生日期**：用于推算年龄。年龄是认知发展的关键变量——同一条行为，3岁宝宝和30岁成人的认知意义完全不同

可选：
- **昵称**：日常称呼，记录观察时可替代名字
- **关系**（默认"未知"）：如"儿子""朋友""自己"，辅助理解观察语境
- **追踪模式**（默认"轻度"）：
   - **深度**：建议每周合成一次，设置 cron 提醒
   - **轻度**：建议每月合成一次，不设 cron

**年龄对认知分析的影响**：

| 年龄段 | 认知特征 | 分析侧重 |
|--------|---------|---------|
| 0-3岁 | 感知运动、客体永久性、语言萌芽 | 身体发展、感知反应、情绪表达、因果关系萌芽 |
| 3-6岁 | 前运算、符号思维、自我中心 | 想象力、社交规则学习、情绪调节、道德感萌芽 |
| 6-12岁 | 具体运算、逻辑思维发展 | 学习策略、社交关系、自我评价、规则内化 |
| 12-18岁 | 形式运算、身份探索、同伴影响 | 价值观形成、决策模式、独立vs依赖、风险倾向 |
| 18-30岁 | 身份巩固、职业探索、亲密关系 | 世界观稳定化、长期决策、压力应对、自我认知 |
| 30-50岁 | 责任期、中年转折、认知成熟 | 优先级排序、代际关系、职业决策、人生意义 |
| 50-65岁 | 经验整合、传承需求、身体变化 | 智慧表达、遗产意识、适应变化 |
| 65岁+ | 回顾整合、身体限制、社会角色转变 | 生命回顾、接受与放下、代际传承 |

画像合成时，系统会根据当前年龄自动调整分析框架的侧重方向和期望基准。

如果用户只说了名字，没有更多信息 → 追问性别和出生日期（这是必填项），关系和模式用默认值。

**Step 2: 创建目录结构**

```
data/[person-id]/
  person.json
  observations/
  signals.jsonl              # 初始化为空（工作记忆，append-only）
  signals-archive/
    consolidating/           # 月度归档
      YYYY-MM-raw.json       # 月度原始信号（完整保留）
      YYYY-MM-summary.md     # 月度归档摘要（聚类合并后的增强版）
    long-term/               # 年度归档
      YYYY-raw/              # 原始信号按月保留
      YYYY-summary.md        # 年度信号摘要（跨月合并，原文摘录全部保留）
    schemas/                 # 图式记忆（永久沉淀）
  profile-history/
```

- `person-id` 生成规则：名字的小写拼音/英文 + 短随机后缀（如 `leo-a3f2`）
- `person.json` 内容：
  ```json
  {
    "id": "[person-id]",
    "name": "[名字]",
    "nickname": "[昵称或null]",
    "gender": "[男/女/其他]",
    "birth_date": "YYYY-MM-DD",
    "relationship": "[关系]",
    "created_date": "YYYY-MM-DD",
    "tracking_mode": "deep/casual",
    "observation_count": 0,
    "last_observation": null,
    "last_synthesis": null,
    "maturity_level": "early",
    "cron_job_id": null
  }
  ```
- 更新 `data/registry.json`，追加新条目。registry.json 格式：
  ```json
  {
    "persons": [
      {
        "id": "[person-id]",
        "name": "[名字]",
        "relationship": "[关系]",
        "created_date": "YYYY-MM-DD",
        "tracking_mode": "deep/casual"
      }
    ]
  }
  ```

**Step 3: 设置 cron（仅深度模式）**

使用 `CronCreate` 创建周期性提醒：
- 深度模式：每周一 9:17（`cron: "17 9 * * 1"`）
- 轻度模式：不设 cron
- `recurring: true`，`durable: true`
- prompt 内容：`"执行 growth-skill Phase 2 信号检查。读取 data/registry.json，对 tracking_mode 为 deep 的追踪对象执行周期性信号检查。如果有显著变化，提醒用户做成长合成。"`
- 将 cron job ID 写入 person.json

**注意**：durable cron 任务有 7 天自动过期机制。如果用户超过 7 天未使用 Claude Code，cron 会消失。Phase 5 追踪总览中会检查 cron 是否过期并提示重新设置。

**完成确认**：
```
已开始追踪 [名字]。直接跟我说观察到的事情就行，比如"今天Leo第一次站起来了"。
[如果深度模式]：每周一早上会提醒你做一次成长合成。
```

### Phase 0B: 记录管理

**删除记录**：

触发：「删除记录 [日期]」或「删掉 [日期] 的记录」

1. 确认要删除的记录（显示日期和内容摘要）
2. 删除 `observations/YYYY-MM-DD.md`
3. 从 `signals.jsonl` 中移除 `observation_dates` 包含该日期的所有信号
4. 如果某信号因删除而 occurrence_count 降为 0，则移除该信号
5. 确认完成：「已删除 [日期] 的记录和相关信号」

**修正记录**：

触发：「修正 [日期]」或「改一下 [日期] 的记录」

1. 读取原记录，展示给用户确认
2. 原文件重命名为 `observations/YYYY-MM-DD.md.superseded`
3. 接收修正后的文本，写入新的 `observations/YYYY-MM-DD.md`
4. 对比原信号和新提取的信号：
   - 新增的信号 → 追加到 signals.jsonl
   - 消失的信号 → 降低 occurrence_count 或移除
   - 变化的信号 → 更新描述，标记 evolving
5. 确认完成：「已修正 [日期] 的记录，信号已更新」

### Phase 0C: 对话纠偏（Phase 1 后置钩子）

**触发条件**（必须同时满足）：
1. 前一轮对话是 Phase 1 的信号提取回应
2. 用户当前输入表达对信号提取的异议：「不对」「撤回信号」「信号有误」「不是这样」「这个判断不对」等

**为什么不是入口分流的独立路径**：「不对」「修正」等是日常高频词，如果作为独立触发词会在无关场景中误判。作为 Phase 1 的后置钩子，只有在信号提取回应之后才激活，彻底消除误触发。

**与 Phase 0B 的区别**：Phase 0B 修正的是原始观察记录（日记原文），Phase 0C 修正的是系统从观察中提取的信号（认知判断）。两者独立运作——原始记录可以是准确的，但系统提取的信号可能偏离观察者的本意。

**执行流程**：

1. **定位信号**：根据上一轮 Phase 1 回应中提到的信号，找到对应的信号
2. **展示确认**：列出涉及到的信号，让用户指出哪条有误
   ```
   最近的信号提取中，以下几条可能需要调整：
   1. [信号类型] "[信号内容]" — 来自 [日期] 的观察
   哪条需要修正？
   ```
3. **用户选择修正方式**：
   - **撤回信号**：追加一行 `status: "removed"` 的标记行到 signals.jsonl（append-only，不删除旧行），同时恢复关联信号的 occurrence_count（如果关联其他观察则保留但减 1）
   - **修正描述**：追加更新后的信号行，标记 `corrected: true`
   - **降级状态**：追加更新后的信号行，如 established → emerging
4. **确认完成**：
   ```
   已修正：[原信号描述] → [新信号描述]
   后续观察会基于修正后的信号继续追踪。
   ```

**设计原则**：
- 纠偏是轻量操作，不需要重新处理整条观察记录
- 纠偏后不自动触发画像合成，避免连锁反应
- 如果同一 `type` 的信号在过去 10 次 Phase 1 中被纠偏 ≥ 3 次，在下次合成时提示：「之前多次修正了 [类型] 类信号，可能需要调整提取策略」

### Phase 0D: 版本回滚

**触发**：用户说「回滚 [名字]」「恢复画像到 [日期]」「回退到 [日期] 的画像」。

**执行流程**：

1. **展示历史版本**：读取 `profile-history/` 目录，列出所有可用的历史画像
   ```
   [名字] 的画像历史：
   1. profile-2026-03-01.md（基于 38 条观察）
   2. profile-2026-02-01.md（基于 25 条观察）
   3. profile-2026-01-01.md（基于 12 条观察）
   要恢复到哪个版本？
   ```
2. **用户确认后执行**：
   - 将当前 `profile.md` 备份为 `profile-[当前日期].md.pre-rollback`（保留在 profile-history/ 中）
   - 将选定的历史版本复制为新的 `profile.md`
   - **信号不回滚**：信号层（signals.jsonl + 归档）保持不变，回滚只影响画像输出。理由是信号是客观的观察记录，不应因画像偏好而改变
3. **确认完成**：
   ```
   已将 [名字] 的画像恢复到 [日期] 版本。
   注意：信号数据未回滚，下次合成时会基于当前全部信号重新生成。
   如果需要彻底回到那个时间点的状态，请告诉我。
   ```

**设计原则**：
- 画像回滚是"软回滚"——只恢复画像文件，不动信号数据
- 如果用户需要"硬回滚"（信号也要回滚），需要额外确认并明确告知风险（会丢失回滚点之后的观察数据）
- 回滚后的画像在下次合成时会被覆盖，回滚本质上是一次性的快照查看

### Phase 1: 记录观察

**触发**：Phase 0 分流到此，或模糊触发确认后。

**Step 1: 确定追踪对象**

- 读取 `data/registry.json` 获取所有追踪对象
- 如果用户明确提到了名字 → 匹配
- 如果只有1个追踪对象 → 默认是该对象
- 如果有多个追踪对象且用户未指定 → 简短确认：「这是关于谁的记录？[列出名字]」

**Step 2: 写入观察文件**

- 文件名：`observations/YYYY-MM-DD.md`（同一天多条记录追加到同一文件，用 `<!-- record-separator -->` 分隔，避免与 YAML frontmatter 的 `---` 冲突）
- Frontmatter：
  ```yaml
  ---
  person: [person-id]
  date: YYYY-MM-DD
  time: HH:MM（可选，如果用户提到具体时间）
  ---
  ```
- 正文：用户提供的自由文本，原样保存，不做任何修改

**Step 3: 轻量信号提取**

读取当前 `signals.jsonl`（工作记忆）+ 所有 `signals-archive/consolidating/*-summary.md` 和 `signals-archive/long-term/*-summary.md` 归档摘要（扫描信号 content 列表做匹配判断，匹配到后再读取完整信号详情）。

对新观察执行轻量提取：
1. **匹配已有信号**：新观察的内容是否与已有信号的描述相关？
   - 匹配 → 该信号 occurrence_count+1，更新 last_seen，检查是否应升级状态
   - 匹配且该信号在归档中 → **再巩固**：从归档召回，stability 保留历史累积值
2. **发现新信号种子**：新观察中是否有 novel 的认知模式？
   - 是 → 创建新信号，status=seed，根据信号类型设置初始 stability
3. **检测矛盾**：新观察是否与某条 established/emerging 信号矛盾？
   - 是 → 创建 contradiction 类型信号，关联原信号 ID
4. **检测里程碑**：新观察是否包含重要人生事件？
   - 是 → 创建 milestone 类型信号

将新/更新的信号追加到 `signals.jsonl`（append-only：更新时追加新行，不修改旧行；读取时按 `id` 去重取最新行）。

**Step 4: 简短回应**

回应原则：像一位细心的倾听者，不是数据分析员。
- 提取 1-2 个最值得关注的发现
- 用自然语言，不要用技术术语（不说"信号置信度0.7"）
- 如果是新追踪对象的早期记录，强调"这只是初步印象"
- 回应末尾自然暗示纠偏入口：「如果上面的判断不太对，直接告诉我就好」

示例回应：
```
记下了。注意到一件事：这是第二次提到"遇到困难先自己试"了，
上一次是在 [日期]。一个模式可能正在浮现。
```

或（早期阶段）：
```
记下了。这是关于 [名字] 的第 [N] 条记录，信息还太少，
暂时不做过多解读。继续记录就好。
```

### Phase 2: 周期性信号检查

**触发**：cron 任务（深度模式每周一次，轻度模式每月一次）。

**执行逻辑**：
1. 扫描该对象的所有信号（工作记忆 `signals.jsonl` + 巩固区 `*-summary.md`）
2. 检查是否有信号状态需要更新（如 emerging 信号已满足 established 条件）
3. 检查是否有遗忘曲线 retention < 0.15 的信号需要标记 forgotten
4. 检查是否有新矛盾或里程碑
5. **如果有显著变化**（信号升级、新矛盾、里程碑）→ 提醒用户："最近 [名字] 有一些值得关注的认知变化，要现在做一次成长合成吗？"
6. **如果无显著变化**→ 静默，不打扰用户

### Phase 3: 画像合成（"睡眠巩固"）

**触发**：用户手动说「演化 [名字]」「更新画像」，或 Phase 2 检测到显著变化后用户确认。

**前置读取**：读取 `references/growth-framework.md` 获取完整方法论（信号类型、遗忘曲线、自适应阈值、矛盾处理、质量自检清单）。

**Step 1: 记忆检索**

从四层记忆中检索该对象的所有认知信号（只读取摘要文件，原始文件仅作保留，不参与合成读取）：

| 记忆层 | 读取路径 | 作用 |
|--------|---------|------|
| 工作记忆 | `signals.jsonl` | 主要数据源 |
| 巩固区 | `signals-archive/consolidating/*-summary.md` | 补充近期模式 |
| 长期记忆 | `signals-archive/long-term/*-summary.md` | 验证持久性 |
| 图式记忆 | `signals-archive/schemas/*.md` | 已确立模型的历史深度 |

**Step 2: 信号归档（"巩固"）**

执行类似人脑睡眠中的记忆整合。归档时**保留原始信号文件不变**，另生成归档摘要文件：

1. **工作记忆容量检查**：如果 signals.jsonl 超过 200 条信号，将 retention 最低的信号移入巩固区
2. **月度归档**：
   - 将上月及更早的信号原始文件**原样复制**到 `signals-archive/consolidating/YYYY-MM-raw.json`（保留全部细节：原文摘录、完整上下文、所有字段）
   - 同时生成归档摘要 `signals-archive/consolidating/YYYY-MM-summary.md`：聚类合并高度相似的信号为增强版（保留所有 occurrence_count 和跨域计数，原文摘录全部保留）
   - 从 signals.jsonl 中移除已归档的信号
3. **显著性标记**：milestone 和 contradiction 信号标记为高显著性，在归档中优先保留
4. **年度转换**：如果当前是1月，将上一年度的 consolidating 归档合并转入长期记忆：
   - 原始文件 `consolidating/YYYY-MM-raw.json` → `long-term/YYYY-raw/` 目录下保留（不压缩）
   - 摘要文件 `consolidating/YYYY-MM-summary.md` → 合并为 `long-term/YYYY-summary.md`，跨月聚类合并，保留信号骨架和全部原文摘录
   - consolidating/ 中已转入 long-term/ 的文件保留不删除

**Step 3: 遗忘计算与再巩固**

对每条信号计算：
```
retention = e^(-t / stability)
effective_weight = significance × retention × recency_boost
```

- retention < 0.15 的信号标记为 `forgotten`
- 检查 forgotten 信号是否被新观察重新匹配（通过内容语义匹配）
- 若匹配 → **再巩固**：从归档召回，写入工作记忆，标记 `reconsolidated_from`，stability 保留历史累积值

**Step 4: 图式沉淀**

检查是否有 established 状态且保持超过6个月的信号：
- 是 → 将该信号提炼为 `signals-archive/schemas/schema-[model-name].md`
- 图式格式：自然语言描述，记录这个认知模式从种子到确立的完整成长轨迹
- **图式记忆不遗忘**，即使原始信号被归档，图式仍然存在

**Step 5: 自适应验证与合成**

根据当前观察总量确定成熟度（early/developing/mature），应用对应的验证阈值。

从通过验证的信号中构建画像：
0. **身份**：综合所有信号和观察记录，用一段话描述这个人在观察者眼中的定位（不是简历，是感受）
1. **心智模型**：从 established 信号中筛选，按 effective_weight 排序，取 top N（N ≤ 对应成熟度的最大模型数）
2. **决策启发式**：从 emerging/established 的 decision_heuristic_seed 中提取
3. **表达模式**：从 expression_fingerprint 信号中归纳
4. **价值观与反模式**：从 value_expression 和 anti_pattern 信号中提取
5. **矛盾与张力**：从 contradiction 信号中分类（时间/领域/本质）
6. **成长时间线**：按时间排列关键认知变化节点
7. **观察盲区**：基于领域覆盖分析
8. **观察者偏差**：基于观察记录的整体倾向分析
9. **诚实边界**：基于数据量和覆盖度的局限标注

**Step 6: 生成画像**

1. 读取 `references/profile-template.md` 获取标准结构
2. 将 Step 5 的结果按模板填入，每个心智模型的「记忆强度」填入其 effective_weight 值
3. 如果已有 profile.md → 先归档到 `profile-history/profile-YYYY-MM-DD.md`
4. 写入新的 `profile.md`
5. 更新 `person.json`：回写 `maturity_level`、`observation_count`、`last_synthesis` 为当前值

**Step 7: 质量自检**

读取 `references/growth-framework.md` 末尾的质量自检清单，逐项检查：
- 模型数量是否适配成熟度？
- 每个模型是否有局限性描述？
- 矛盾是否被追踪而非"解决"？
- 置信度是否不超过成熟度上限？
- 观察盲区和偏差是否已标注？
- 语气是否像传记笔记而非数据报告？

不通过 → 修正后重新检查。最多2轮。

**Step 8: 输出结果**

向用户展示合成摘要：
```
[名字] 的认知画像已更新。

📊 当前状态：[成熟度]，[N]条观察，[N]个心智模型

✨ 本次变化：
- [新出现的模型/信号升级/矛盾发现]

⚠️ 值得关注：
- [需要留意的趋势或盲区]

完整画像见 data/[person-id]/profile.md
```

### Phase 4: 成长复盘

**触发**：用户说「成长报告」「复盘」「[名字]的成长」「成长轨迹」。

**Step 1: 读取历史**

读取 `profile-history/` 目录下的所有历史画像，按时间排序。
同时读取工作记忆和归档摘要中的信号变化（信号升级/降级、新矛盾、新里程碑），补充画像快照之间未反映的细节。

**Step 2: 生成成长叙事**

对比相邻版本，识别变化：

| 变化类型 | 检测方式 | 叙事方式 |
|---------|---------|---------|
| 新模型 | 上一版没有，这一版出现 | "一个新的认知模式正在形成..." |
| 模型升级 | emerging → established | "XX 从初步印象成长为确立的认知模式" |
| 模型演化 | 描述或应用范围变化 | "XX 的内涵发生了变化：以前是...，现在是..." |
| 模型休眠 | established → dormant | "XX 最近不再出现了，可能是一次阶段性的行为" |
| 新矛盾 | 新出现的 contradiction | "一个有趣的转变：以前...，现在..." |
| 矛盾解决 | 时间矛盾中新模式稳定替代旧模式 | "之前的矛盾有了答案：XX 最终..." |
| 置信度变化 | effective_weight 显著升降 | "对 XX 的信心增强/减弱了" |

**Step 3: 输出格式**

输出应该像一位传记作家写给读者的信，不是数据仪表盘。

```
# [名字] 的成长记录

## 成长概览
从 [起始日期] 到 [结束日期]，[名字] 的认知世界经历了以下变化。

## 最早期的信号（[时间段]）
那时候，[名字] 的行为模式可以用一个词概括：**[关键词]**。
[N]条记录中，有[M]条涉及[描述]。

## 第一个认知转折（[时间点]）
[具体观察引用]
这是 **[模型名]** 的种子信号。到 [时间点]，这个模式在 [N] 个不同领域反复出现，
升级为已确立的心智模型。

## 正在浮现的模式（最近）
[具体观察引用]
这个信号目前还是种子状态，需要更多观察来确认。

## 你可能忽略的维度
[基于盲区分析的建议]

## 整体趋势
[一段总结性的成长叙事，50-100字]
```

**深度复盘**：用户说「详细复盘」或「成长轨迹」时，输出上述完整格式。
**快速复盘**：用户说「[名字]的成长」时，只输出"成长概览"和"正在浮现的模式"两个 section。

### Phase 5: 追踪总览

**触发**：用户说「看看状态」「所有人」「追踪列表」。

读取 `data/registry.json`，展示所有追踪对象：

```
📊 成长追踪总览

| 名字 | 关系 | 观察数 | 成熟度 | 上次记录 | 上次合成 | 状态 |
|------|------|-------|--------|---------|---------|------|
| Leo | 儿子 | 47 | developing | 04-15 | 04-14 | ✅ 活跃 |
| 奶奶 | 奶奶 | 8 | early | 04-10 | 03-01 | ⚠️ 需要关注 |
```

**提醒规则**：
- 超过14天没有新记录 → 标注"需要关注"
- 超过30天没有新记录 → 标注"可能已停止追踪"，建议确认是否继续
- 上次合成距今超过对应模式的周期（深度>7天，轻度>30天）→ 建议做一次合成
- 深度模式且 cron_job_id 对应的 cron 已过期（>7天）→ 标注"定时提醒已过期"，建议重新设置 cron

## 特殊场景

### 冷启动（只有1-3条记录）

- 不生成 profile.md（数据太少，强行生成会误导）
- Phase 1 的回应中明确说"信息还太少，继续记录就好"
- 当观察数达到5条时，提示用户可以尝试第一次合成

### 多人追踪同一事件

如果一条观察涉及多个被追踪对象（如"Leo和妹妹一起搭积木"）：
- 询问用户要记到谁的记录下，还是分别记录
- 分别记录时，各自提取独立的信号

### 观察者记录自己

当追踪对象是观察者本人时：
- 特别注意自我认知偏差——人往往高估自己的某些特质、忽略盲点
- 在 Phase 3 合成时，观察者偏差 section 要重点标注
- 可以追问："身边的人怎么看你的这个特点？"

### 长期未使用后恢复

如果用户长时间未使用 growth-skill 后重新开始：
- 检查 registry.json，展示上次状态
- 如果有未归档的信号，先执行一次归档再继续
- 不需要重新追踪，所有历史数据都在

## 品味守则（速查）

遇到判断困难时回看。具体量化标准见 `references/growth-framework.md`。

| 原则 | 一句话 |
|------|--------|
| 变化 > 固定 | 改变行为的地方比一直坚持的更有信息量 |
| 矛盾 > 一致 | 内在矛盾是成长的核心证据 |
| 少即多 | 3个深刻的模型远好于10个浅薄的原则 |
| 诚实 > 完美 | 宁可标注局限的60分画像，也不要编造的90分画像 |
| 追踪 > 判断 | 记录"发生了什么"，不要急着下结论 |

### 绝不做的事

- 在数据稀疏时强行提取心智模型
- 把一次观察包装成"已确立的认知模式"
- 忽略矛盾（矛盾=成长信号）
- 在不知道的时候假装知道
- 用技术术语回应用户（"信号置信度0.7" → "一个模式可能正在浮现"）
- 替观察者判断他们看到了什么

## 最后

growth-skill 不创造认知，它发现认知。
最好的成长记录不是每条都精彩，而是持续记录后回头看时，能看到那些当时没注意到的轨迹。

开始追踪吧。
